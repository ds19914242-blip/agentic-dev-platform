I've reviewed the sources page, the test endpoint, the storage layer, and the type definitions. Notably, `RssSource` already carries `lastStatus`/`lastError`/`lastItemCount`/`lastFetchedAt` fields, the UI already renders `lastStatus` inline, but the `/api/rss/test` endpoint does **not** persist these results — so today the status is shown per-toast only and never aggregated. That context shapes the spec below.

# Feature Specification

## Summary

Маленький тестовый блок статуса источников — компактный сводный виджет на странице «Источники» (`app/sources/page.tsx`), показывающий агрегированное состояние RSS/Telegram/Reddit-лент: сколько источников активно, сколько в последней проверке вернули `ok` / `empty` / `error`, и когда была последняя проверка. Блок опирается на уже существующие, но сейчас не заполняемые поля `lastStatus`, `lastItemCount`, `lastFetchedAt` в `RssSource`. Цель — дать пользователю мгновенный обзор «здоровья» источников без необходимости проверять каждый по отдельности.

Объём намеренно минимальный («маленький тестовый блок»): один сводный блок в верхней части списка источников плюс минимальная связка, чтобы кнопка «Проверить» сохраняла результат проверки в источник.

## User Stories

- Как аналитик, я хочу видеть в одном месте, сколько моих источников рабочие, а сколько со сбоями, чтобы быстро понять состояние подборки.
- Как аналитик, я хочу, чтобы результат нажатия «Проверить» сохранялся, чтобы статус оставался виден после перезагрузки страницы, а не исчезал вместе с тост-уведомлением.
- Как аналитик, я хочу видеть время последней проверки источника, чтобы понимать, насколько свежи данные о статусе.
- Как аналитик, я хочу одним действием запустить проверку всех активных источников, чтобы обновить сводку без ручного перебора (опционально, см. Out of Scope).

## Acceptance Criteria

- [ ] На странице «Источники» над списком отображается компактный блок статуса с агрегированными счётчиками: всего источников, активных, и по последнему статусу — `ok`, `empty`, `error`, «не проверялись».
- [ ] Счётчики вычисляются из полей `lastStatus` / `isActive` существующих источников и обновляются после загрузки списка (`load()`).
- [ ] При нажатии «Проверить» на источнике результат (`lastStatus`, `lastItemCount`, `lastError`, `lastFetchedAt`) сохраняется в хранилище и виден после перезагрузки страницы.
- [ ] Блок показывает время последней проверки по подборке (например, самое свежее `lastFetchedAt`) в человекочитаемом виде.
- [ ] Если источников нет, блок либо скрыт, либо показывает нейтральное пустое состояние без ошибок.
- [ ] Цветовая индикация согласована с существующей в списке (зелёный — ok, янтарный/серый — empty, розовый/красный — error).
- [ ] Блок не ломает текущую вёрстку (адаптивная сетка `lg:grid-cols-3`) и работает при состоянии загрузки (`sources === null` — shimmer/скелет).
- [ ] Никакие чувствительные данные (URL с токенами, тексты ошибок с внутренними деталями) не раскрываются сверх того, что уже показано в списке.

## Scope

- Новый презентационный блок статуса в `app/sources/page.tsx` (или маленький выделенный компонент, например `components/SourceStatusPanel.tsx`).
- Агрегация статусов на клиенте по уже загруженному массиву `sources`.
- Персистенция результата проверки: расширение `/api/rss/test` (или дополнительный вызов `PATCH /api/rss/sources/[id]`) так, чтобы `lastStatus`, `lastItemCount`, `lastError`, `lastFetchedAt` записывались в источник через `rssSourceStorage.update`.
- Использование существующих типовых полей `RssSource` без изменения схемы хранения.

## Out of Scope

- Кнопка «Проверить все» / массовая фоновая проверка (можно вынести в отдельную последующую задачу).
- Фоновое расписание / cron-проверки доступности источников.
- История статусов во времени, графики, метрики аптайма.
- Изменение логики сбора (`lib/rss/collect.ts`, fetchFeed/telegram/reddit) и форматов данных.
- Уведомления (email/Telegram) о падении источника.
- Многопользовательское разграничение статуса сверх уже существующего `ownerId`-скоупинга.

## Risks

- Поля `lastStatus` и т.д. сейчас объявлены, но не заполняются: если их не начать персистить, блок всегда покажет «не проверялись». Нужно аккуратно увязать проверку и сохранение.
- `/api/rss/test` сейчас не привязан к id источника (принимает только url/sourceType). Для персистенции потребуется передавать id либо делать отдельный PATCH — риск рассинхрона, если проверяемый url отличается от сохранённого.
- Telegram/Reddit-проверки зависят от внешних публичных страниц и могут давать нестабильный `empty`/`error`, из-за чего сводка будет «дёргаться».
- Гонки при параллельных проверках нескольких источников (последовательные `update` поверх одного JSON-файла в `lib/storage/rss.ts`) могут терять записи — учитывать при реализации.
- Семантика статуса `empty` (доступен, но пуст) не должна считаться ошибкой в сводке, иначе вводит в заблуждение.

## Manual Verification Scenarios

- Открыть `/sources` с несколькими источниками разных типов (rss/telegram/reddit): блок статуса отображается, счётчики «всего/активных» совпадают с реальным списком.
- Нажать «Проверить» на рабочем RSS-источнике → дождаться тоста `OK` → перезагрузить страницу: у источника сохранён статус `ok` и счётчик `ok` в блоке увеличился.
- Нажать «Проверить» на заведомо битом URL → статус `error` сохранён, отражён в блоке (красный счётчик), и текст ошибки виден в строке источника.
- Проверить источник, который доступен, но пуст → статус `empty` учтён отдельно, не как ошибка.
- Отключить источник (toggleActive) → счётчик «активных» уменьшается, общий — нет.
- Открыть `/sources` без источников → блок скрыт или показывает корректное пустое состояние.
- Открыть страницу при медленной сети (состояние `sources === null`) → блок показывает скелет/shimmer, не падает.
- Проверить, что время «последней проверки» обновляется после новой проверки.

## Decomposition Notes

Позже разбить на бэклог-задачи примерно так:

1. **Backend / персистенция статуса** — увязать проверку источника с сохранением `lastStatus`/`lastItemCount`/`lastError`/`lastFetchedAt`. Решить между двумя подходами: (а) расширить `POST /api/rss/test` приёмом `id` и вызовом `rssSourceStorage.update`, либо (б) оставить `test` чистым и добавить клиентский `PATCH /api/rss/sources/[id]` после проверки. Принять решение в дизайн-задаче. Включить защиту от гонок при записи JSON.
2. **UI-компонент блока статуса** — `SourceStatusPanel` (агрегация из массива `sources`, счётчики, цветовая индикация, пустое/loading-состояние), встроенный в `app/sources/page.tsx`. Зависит от существующих полей, может вестись параллельно с задачей 1 на мок-данных.
3. **Интеграция и состояние** — связать `test()`/`load()` так, чтобы после проверки блок пересчитывался; отобразить время последней проверки.
4. **(Опционально, отдельной задачей)** — кнопка «Проверить все активные» с последовательным/ограниченно-параллельным запуском и обновлением сводки.
5. **Проверка/QA** — ручные сценарии выше; при наличии — лёгкий компонентный тест агрегации статусов.

Ключевая развилка для дизайн-фазы — пункт 1 (где и как персистить статус), так как от него зависит контракт API и форма данных для блока.
